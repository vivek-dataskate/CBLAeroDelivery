## What Went Well

### Architecture Patterns (Biggest Win)

- **Repository pattern (Story 1.8):** Dual persistence (Supabase + in-memory) established as gold standard. Reused in every subsequent story. Enabled fast test iteration without mocking Supabase.
- **AI inference service (Story 1.9):** Centralized Anthropic SDK usage through `modules/ai/`. 4-file pattern (client, registry, service, index) became the service template.
- **withAuth middleware (Story 1.10):** Consolidated ~15-line inline auth block from 17 routes into single higher-order function. Future routes inherit auth+audit+tenant isolation by default.
- **Content fingerprint gate (Story 1.11):** 5 ingestion paths covered (PDF, OneDrive, email, CSV, ATS) with batch optimization (O(1) Set lookups for 10K+ rows).

### Security-First Approach

- Server-authoritative enforcement applied consistently across all 12 stories
- Proxy envelope (Story 1.1) prevented cross-tenant bypass vulnerabilities across all downstream stories
- Audit trail discipline evolved from simple denial logging → admin action stream → step-up events → compliance evidence → client confirmation events → cost tracking
- Tenant isolation deepened incrementally without regressions

### Adversarial Review Process

- Dual-model review (Opus builds, Sonnet reviews) caught ~40% more issues than self-review
- Story 1.9a: Sonnet caught double-filter budget logic, aggregation grouping bug, dead null check
- Story 1.10: Caught dead `requireStepUp`/`requireFreshAuth` options that silently did nothing
- Story 1.11: 17 findings including field name typo that would have broken all Ceipal sync

### Pattern Reuse Velocity

- Story 1.8's repository pattern → 1.9's service pattern → 1.9a's dashboard → 1.10's middleware → 1.11's fingerprint gate
- Each story built on the architectural conventions of previous stories
- Token-level cost tracking (1.9) provided complete data foundation for cost dashboard (1.9a) on day one

## What Was Challenging

- **Test-mode guard omissions:** `clear*ForTest()` functions missing `shouldUseInMemoryPersistenceForTests()` guard — same MEDIUM finding in Stories 1.8, 1.9, 1.11
- **Silent error handling:** Supabase `.error` not checked in Stories 1.8 and 1.11 — led to silent data loss (HIGH findings)
- **Field name inconsistencies:** `.ceipal_id` vs `.ceipalId` camelCase/snake_case mismatch in Story 1.11 (HIGH — would break all ATS sync)

## Key Lessons Learned

1. **Architecture patterns are the highest-leverage investment.** Repository, middleware, and service patterns established in Epic 1 will accelerate every future epic. The upfront cost is repaid many times over.

2. **Dev standards must document enforcement patterns explicitly.** Convention-by-example doesn't scale — when the same mistake appears in 3 consecutive stories, the fix is documentation, not developer discipline.

3. **Scope changes need formal management.** 71% scope growth without a change proposal leads to tracking gaps, incomplete retrospectives, and hidden complexity. Update epic definitions and sprint plans simultaneously when scope changes.

4. **Dual-model adversarial review is the highest-ROI quality gate.** Different models catch different classes of bugs. Using a different model for review vs. implementation catches logic bugs, dead code, and integration issues that self-review misses.

5. **Security-sensitive confirmation flows require three properties together:** identity-bound token, one-time use, and payload intent binding (discovered in Story 1.7, reusable pattern).

6. **Compliance gates should degrade predictably** — avoid accidental 500 paths for known policy failures (Story 1.6 review finding).

7. **Runtime config dependencies must be made explicit** in deployment playbooks to reduce first-run production incidents (Story 1.6 lesson).
